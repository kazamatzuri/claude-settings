# Ready Ticket Requirements

## Required Sections

### 1. Problem Statement
- What problem are we trying to solve?
- Why is this a problem? To whom does it matter?

### 2. Acceptance Criteria
- Written as outcomes (not tasks)
- What does "done" look like?
- Focus on value delivered

### 3. Implementation Details
- How the work will be done (suggested approach)
- GitHub permalinks to prior art or target code
- Links to additional context (Slack, examples)

## Required Fields

- **Labels**: Product area, repositories
- **Priority**: Highest/High/Medium/Low/Lowest
- **Blockers/Relations**: Links to dependent stories

### Priority Guide
| Priority | Use For |
|----------|---------|
| Highest | OKR work, critical bugs |
| High | Bugs, committed non-OKR work |
| Medium | Impactful but not committed |
| Low | Nice to have, rainy day |
| Lowest | Very little customer impact |

## Validation for Ready

Before marking ready, verify:
- [ ] Problem statement is clear (anyone in engineering can understand)
- [ ] Acceptance criteria define outcomes
- [ ] Priority set appropriately
- [ ] Labels added
- [ ] Blockers identified

## Questions to Ask

1. "What problem does this solve? For whom?"
2. "How will we know it's done?"
3. "What priority - is this OKR/committed work?"
4. "Any blockers or dependencies?"
